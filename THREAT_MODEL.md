# Threat Model

Sensitive data lives in three places in this system: Authority credentials (`secrets.json` / shared HMAC secret), the transmitted manifest payloads (crew identity and passport data), and the filing outcomes (`results.json`, future audit logs, and Authority acknowledgments). In this case study the shared secret is read from disk because that is how the mock is packaged, but the code isolates secret loading in `direct_filer/config.py` so the secret source can be replaced with a secret manager or environment injection without touching the signing logic.

The highest-risk path is request signing. A security reviewer should see that the secret is never logged, never embedded in request payloads, and only used to derive the `X-Crescent-Signature` header over the spec-defined canonical string. The code keeps the secret in process memory only long enough to construct the HMAC. In production, I would move it into a dedicated secret store, rotate it regularly, and prevent local development fixtures from ever sharing namespace with production credentials.

Crew/passport data appears in scenario files and in transmitted manifests, so audit controls matter. A reasonable audit trail would capture: manifest ID, scenario/source identifier, build timestamp, rule-engine outcomes, Authority receipt ID, final acknowledgment status, and the exact canonical payload hash submitted. I would avoid storing full payloads in general application logs; if payload retention is required for compliance, they should be encrypted at rest and access-controlled separately from routine logs.

Primary abuse cases are secret exfiltration, payload tampering, replay of signed requests within the clock tolerance window, and unauthorized access to PII in filings. Concrete next controls would be TLS enforcement, strict outbound network allow-listing to Authority hosts, redaction of PII in logs, short-lived credentials for runtime access to the secret, and append-only audit records for every transmission and acknowledgment poll.
